PHASE 00

Decisions & accounts

Planned

Everything downstream depends on these three calls being made first.

01

Build with Expo (React Native), not Capacitor or two native codebases. Capacitor ships your existing web pages inside a wrapped browser — cheapest, but it reads as a website, not an app. Two separate Swift + Kotlin apps get the most "native" feel but roughly doubles the build and maintenance cost. Expo compiles to real native views on both platforms from one TypeScript codebase, and its cloud build service (EAS) means you never need a Mac to ship the iOS build.See the comparison below — this is the one decision the rest of the guide assumes.

02

Enroll in the Apple Developer Program — required for TestFlight, App Store distribution, and real push certificates. Individual or organization account both work; organization looks more credible on the App Store listing but takes longer to verify (D-U-N-S number).

03

Create a Google Play Console account — one-time fee, no renewal.

04

Create a free Expo (EAS) account. The free tier's monthly build allowance is enough for a project shipping every few weeks; you only need to pay if you're iterating on native builds daily.

Recommended

Expo / React Native

Real native components, one codebase, cloud builds. Gets the premium feel with a deliberate polish pass (Phase 11) at native-app cost.

Cheapest

Capacitor wrapper

Wraps the existing Next.js pages in a native shell. Fastest to ship, but feels like a website in a frame — works against the "feels expensive" goal.

Priciest

Native Swift + Kotlin

Best possible platform feel, but two codebases, two skillsets, roughly 2x the time and ongoing maintenance.

PHASE 06

Cliques — make one, join several

Shipped

A clique is a small group of friends who opted in to find each other. Anyone can start one, name it, and invite people in — and belong to as many at once as they like. The group persists across events; the live location-sharing that powers WYA does not.

Real code, mock backend. Every screen below is built and click-tested — create, search-invite, accept/decline, expand-in-place, Manage/Leave. It runs on a local Context store (store/AppStore.tsx) standing in for the tables and routes in steps 01 and 06; Phase 03/04 swap that store for the real Supabase calls without touching any screen.
01

Three tables, not two — cliques stop being a single hardcoded group: cliques (id, name, creator_id, hero_photo_path, created_at), clique_members (clique_id, user_id, color_hex, joined_at), and clique_invites (id, clique_id, inviter_id, invitee_id, status, created_at) where status is pending / accepted / declined.

02

Each member still gets a fixed color from a six-swatch palette, assigned in join order: coral #FF6B5B · amber #FFB648 · seafoam #4DDDC9 · sky #3DC7F0 · periwinkle #7C93FF · violet #B98BFF — chosen to sit clear of the brand magenta and the "confirmed" status green so nothing on the radar gets misread as a system color. A clique past six members cycles the palette.

03

Invites are search, not links. The creator searches app accounts by name or Instagram handle — the same lookup WYA's search reuses — and taps Invite. The invite sits pending until the other person accepts or declines it themselves, from a card at the top of their own Cliques feed. Accepting inserts the clique_members row; declining just closes the invite out. Nobody lands in a clique without saying yes, and no separate invite-link system is needed since everyone already has the app installed to ever show up on the radar.

04

The Cliques screen is an Instagram-style feed, one card per clique: a square hero image, a facepile of member avatars plus the wya? button underneath, and the clique's name and member count as a caption. The hero is a photo the creator uploads, or — when they haven't — a mosaic collage tiling each member's avatar color/initial (or real photo, once uploaded) into a grid. The mosaic is rendered client-side from current membership, not a generated image file, so it's always accurate and costs nothing to keep up to date.

05

Tapping a card expands it in place — no navigation, no new screen — revealing the full member list right there in the feed. The panel forks on role: the creator gets a Manage Clique section to remove any member or add more through the same search-invite flow; everyone else gets a single Leave Clique action instead.

06

Five routes: POST /api/cliques (create, optional hero upload), POST /api/cliques/:id/invites (search-invite), POST /api/clique-invites/:id/respond (accept/decline), DELETE /api/cliques/:id/members/:userId (creator removes anyone, a member removes themselves), GET /api/cliques (list mine, each with members, hero, and any pending invites).

New spend this phase$0 — same Supabase project, mosaic hero rendered client-side
PHASE 07

WYA — the proximity radar

Partial

This is the one phase in the whole guide with genuine engineering risk. Everything else here is integration work; this is new math, worth prototyping standalone before it's wired into the full app.

UI shipped, math still placeholder. The radar, search, and precision-finder screens are real and now multi-clique aware (scoped by cliqueId, working for both Night Owls and any other clique). Steps 01–06 below — the actual GPS, compass, and Realtime Presence engineering — haven't started; distanceMeters/bearingDeg are still the hardcoded placeholder values called out in data/clique.ts.
01

Pick GPS + compass over Bluetooth/UWB. Apple's Nearby Interaction framework gives precise distance-and-direction on iPhone 11+, the same tech behind AirTag's Precision Finding — but it has no real Android counterpart, and this app needs to feel equally premium on both. GPS-plus-compass is cross-platform, needs no extra hardware, and the festival is an open-air venue where GPS is accurate to 5–10m — plenty for a radar that shows roughly which way and roughly how far, which is the actual brief.The honest tradeoff: below ~5m, GPS noise alone can make an arrow jitter. That's the real problem to solve, in step 4 — not the choice of positioning technology.

02

Transport — while WYA is open, share foreground-only GPS (expo-location, one fix every 2–3 seconds) over a Supabase Realtime Presence channel scoped to the clique (clique:<id>). Presence is built exactly for "who's in this room and what's their live state" — zero new infrastructure, same project as everything else.

03

The math — haversine distance and bearing between your last fix and each clique-mate's, then subtract your own phone's compass heading (expo-location's heading API) so the arrow always points at the friend regardless of which way you're facing. Same trick behind Find My's Precision Finding arrow.

04

Fixing the jitter — below ~15m, blend in Bluetooth RSSI as a coarse confidence signal (nearby-only, no direction on its own), and smooth the bearing with a rolling average instead of redrawing on every raw fix. A jumpy arrow is the one thing that would make this feel cheap instead of expensive — it's worth more polish time than any other single screen in the app.

05

The radar — a circle at center for you; each clique-mate as a colored arrow, placed nearer the center as they get closer and further toward the edge as they get farther, brightening as the distance shrinks. Below the point GPS and RSSI agree you're within a few meters, the arrow flashes and a haptic success pulse fires — found.

06

Privacy, non-negotiable — off by default. Opening WYA prompts location permission with a plain-language explainer, defaults to “Share for tonight only” (auto-expiring at EVENT_END), and shows a persistent “Sharing until 09:00” banner with a one-tap stop while it's active. Nothing is ever written to a table — only the latest fix per person lives in the Realtime channel's ephemeral presence state.

This is the feature to build and test in isolation first — two phones, one clique, before it's wired into the rest of the app. Everything else in this guide is proven integration work; this is the one piece that needs its own spike.
New spend this phase$0 — Supabase Realtime is already part of the plan you have
PHASE 08

Payments — bKash wallet, cards via SSLCommerz, plus add-ons

Partial

This replaces the site's current "we'll send payment instructions" step with real in-app checkout, then layers on the shuttle add-on and a separate accommodation booking. It's the one place this guide adds meaningful new backend work.

Screens and state machine shipped, gateways not. Registration → approval → Payment → shuttle toggle → Tickets, and the whole accommodation reserve → approve → pay cycle plus the Reservations section, are real and click-tested end to end. Approval is currently a local timer (~3.5s) standing in for admin review, and the two Pay buttons just flip a local paid flag — steps 01–05 below (the actual bKash/SSLCommerz integration) haven't started, and can't until Phase 03/04 give the app a real backend to verify against.
01

Decide where payment sits in the existing status flow. Recommended default: keep ID review first — a ticket only becomes payable once admin approves it, exactly as today — then payment moves it straight to confirmed. This avoids refunding rejected buyers. The alternative, charging at checkout before ID review, converts better but means building a refund path for anyone later rejected.Flag if you'd rather charge upfront — it changes the order of steps below, not the integrations themselves.

02

bKash — integrate the Tokenized Checkout API: your backend requests a grant token, creates the payment, and gets back a hosted bKash payment URL. Neither bKash nor SSLCommerz publishes a public React Native SDK, so this one screen opens that URL in a secure in-app WebView — the same pattern Pathao, Chaldal, and foodpanda use for their own bKash checkout. It doesn't undercut the native-app goal; it's one screen out of the whole product.

03

Cards, via SSLCommerz — same hosted-checkout shape: your backend opens a session and gets a GatewayPageURL, shown in the same WebView. Wire up SSLCommerz's IPN (a server-to-server webhook) rather than trusting the redirect alone — it's the reliable half of the confirmation.

04

Never trust a client-side "success" redirect by itself. Both integrations must re-verify server-side before marking a ticket paid — bKash via its Query Payment call, SSLCommerz via Order Validation or the IPN payload. A deep link or screenshot is not proof of payment.

05

Add one table and four routes:

-- new table
payments (
  id, ticket_id, user_id,
  provider          -- 'bkash' | 'sslcommerz',
  provider_txn_id, amount, currency,
  status            -- 'initiated' | 'success' | 'failed' | 'refunded',
  raw_payload jsonb, created_at
)

// new routes
POST /api/payments/bkash/create        // grant token → create payment
POST /api/payments/bkash/callback      // execute + query, server-verified
POST /api/payments/sslcommerz/create   // open session → GatewayPageURL
POST /api/payments/sslcommerz/ipn      // server-to-server confirmation
Prerequisite, not a bug: production merchant accounts with both bKash and SSLCommerz require a registered Bangladeshi business (trade license) and settlement bank account. Sandbox credentials for both are free and available immediately, so development isn't blocked — but start the merchant application in parallel with Phase 00, since approval can take longer than the build itself.
New spend this phase$0 to integrate — gateways take a per-transaction % (~1.5–2% bKash, ~2.5–3% cards), not a fixed fee

Shuttle stays an add-on on this same Payment page; accommodation does not — it runs its own reserve → approve → pay lifecycle entirely separate from the ticket.

06

Shuttle is a straight port of the site's existing checkbox — same Tk 800 round-trip, same copy — now a toggle row on the ticket's Payment page with the total recalculating live. Whether it also stays collected at registration, or moves here exclusively, is still open.

07

Accommodation is its own registration, not a ticket add-on — keyed by event_id so a second event slots in without a migration, with its own pending / approved / rejected status and its own payment. A guest can pay for their ticket without ever touching accommodation, or vice versa; neither blocks the other.

08

Gated inside the event card. "Need Accommodation?" only appears once that guest's own ticket registration for that event reads approved — before that, the card shows just registration status, with nothing to reserve. Tapping it opens the same reserve → pending → approved-and-pay states as the ticket flow, scoped to that one reservation.

09

The Tickets tab's new Your Reservations section (Phase 05) lists one row per accommodation booking across every event the guest is attending, each with its own status pill, linking back into that reservation's current state.

10

One new table:

-- new table
accommodation_reservations (
  id, event_id, user_id,
  status            -- 'pending' | 'approved' | 'rejected',
  tier, price, currency,
  reference_code, created_at
)

-- payments table (above) gains one nullable column
-- so a row can settle either a ticket or a reservation, not both
payments.reservation_id  -- nullable, references accommodation_reservations(id)

Confirm whether it's one flat option or several tiers, and whether it's capacity-limited, before wiring real pricing into it.
